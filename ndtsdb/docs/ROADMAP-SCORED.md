# ndtsdb Roadmap 任务评分与优先级

**评分时间**: 2026-02-18  
**评分人**: 4号（策略开发）  
**评分维度**:
- 业务价值 (0-10): 对当前业务的直接价值
- 技术紧迫性 (0-10): 不解决是否会阻塞其他工作  
- 实现复杂度 (0-10): 难度越高分越低（反转计分）
- 资源依赖 (0-10): 依赖外部资源程度（1号/第三方）

**综合分公式**: `业务价值×0.4 + 技术紧迫性×0.3 + (10-实现复杂度)×0.2 + (10-资源依赖)×0.1`

## 📊 详细评分说明

### 🔴 P0 (综合分 ≥ 7.5) - 立即执行

#### 1. 压缩率实测验证 (8.5分)
- **业务价值 9**: 压缩率直接影响存储成本，Gorilla 标称70-90%但实际仅20-30%，需验证并优化
- **技术紧迫性 9**: 阻塞 Phase 1 验收，影响生产环境部署决策
- **实现复杂度 3**: 已有虚拟市场数据生成器，只需跑测试
- **资源依赖 2**: 内部资源，无需外部依赖
- **行动**: 立即执行，2天内输出报告

#### 2. 金融语义 SQL (7.9分)
- **业务价值 9**: 差异化核心，金融场景刚需
- **技术紧迫性 8**: 支撑 quant-lib/quant-lab 金融策略开发
- **实现复杂度 4**: SQL 解析器已成熟，扩展函数较简单
- **资源依赖 3**: 需我（4号）设计金融语义接口
- **行动**: 与压缩测试并行开发

#### 3. 回放稳定性测试 (7.6分)
- **业务价值 8**: 3000+ 品种回放是核心卖点
- **技术紧迫性 8**: 9号实盘依赖此功能稳定性
- **实现复杂度 4**: 已有 MmapMergeStream，需长时间运行测试
- **资源依赖 2**: 内部资源
- **行动**: 挂起后台长期运行测试，监控内存/稳定性

### 🟡 P1 (综合分 6.5-7.5) - 本周内

#### 4. 高并发压力测试 (7.1分)
- **业务价值 7**: 验证生产环境承载能力
- **技术紧迫性 8**: 多品种并发写入需验证
- **实现复杂度 5**: 需设计并发测试方案
- **资源依赖 3**: 内部资源
- **行动**: 1-2天内完成，与压缩测试并行

#### 5. 错误处理规范化 (6.7分)
- **业务价值 7**: 提升代码质量，减少线上故障
- **技术紧迫性 6**: 长期技术债，非阻塞
- **实现复杂度 4**: 重构 throw/return 模式
- **资源依赖 2**: 内部资源
- **行动**: 交给 007（测试组）顺便做

### 🟢 P2 (综合分 5.5-6.5) - 本月内

#### 6. mmap/Zero-Copy 深度验证 (6.2分)
- 大文件场景验证，当前小文件场景已够用
- 1号评估 CLI 时可能涉及，可合并

#### 7. zstd/lz4 压缩集成 (6.1分)
- 压缩率优化，但 Gorilla 已可接受
- 如压缩率测试不达标，提升为 P1

#### 8. ndtsdb-cli (5.8分)
- 重要但不紧急，等待 1号评估结果
- 如 1号认为可行，提升为 P1

### 🔵 P3 (综合分 < 5.5) - 未来规划

- WAL 事务支持、SQL 复杂 JOIN 优化、WebAssembly
- 综合分较低，资源投入产出比不高

### 💤 按需触发 / ⚪ 未来

- 压缩算法 C 实现：仅当性能瓶颈时
- 边缘云同步、嵌入式 AI：Phase 2 后期再看
- 分布式系列：Phase 3，当前不投入

## 📈 评分方法论

### 为什么这样评分

1. **Phase 1 任务高分**: 当前最紧迫，阻塞后续所有工作
2. **验证类任务优先**: 不确定性高，需尽早验证假设
3. **分布式任务低分**: 当前单机足够，过度工程
4. **CLI 中等分**: 重要但不紧急，依赖 1号评估

### 评分争议点

- **zstd/lz4 (6.1分)**: 如压缩率实测不达标，应提升为 P0
- **ndtsdb-cli (5.8分)**: 如 1号评估快速可行，应提升为 P1
- **错误处理 (6.7分)**: 技术债，但非阻塞，可延后
